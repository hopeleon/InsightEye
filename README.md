# InsightEye

[English Version](./README_EN.md)

**InsightEye** 是一个面向智能眼镜与面试辅助场景的候选人分析原型系统。  
它将原始面试文本转换为一个更适合面试官快速决策的界面：候选人风格、风险信号，以及下一步最值得追问的问题。

当前版本聚焦于一个 **Text-First MVP**：

- 输入：带角色标记的完整面试文本
- 处理：面试轮次解析、原子 NLP 特征提取、DISC 导向的行为风格推断
- 输出：面向面试官的决策优先界面

这个仓库不是一个通用的分析展示 Demo。  
它被设计成一个 **面试辅助系统**，核心目标是降低阅读成本、提升面试判断效率。

---

## 项目能力

对于如下输入：

```text
面试官：请你介绍一个推进难度比较高的项目。
候选人：……
面试官：你当时是怎么判断方向正确的？
候选人：……
```

InsightEye 当前可以：

- 识别面试官与候选人的发言轮次
- 推断大致的岗位语境与问题类型
- 从候选人回答中抽取原子语言特征
- 估计 DISC 行为风格倾向
- 识别低细节、话术包装、真实性存疑等风险信号
- 为面试官生成下一步追问建议
- 将结果组织为“决策优先”的界面，而不是一整页原始分析文本

---

## 产品定位

InsightEye 是一个 **面试决策支持层**，而不是人格诊断系统。

系统的输出重点放在：

- **行为倾向**
- **表达风格**
- **证据质量**
- **面试引导**

不应将其理解为：

- 临床人格评估
- 自动化淘汰工具
- 对面试官判断的替代

## 系统架构

当前链路保持“简单、可解释、易验证”的设计原则。

```text
面试原始文本
        ↓
角色解析 / 问答映射
        ↓
原子 NLP 特征提取
        ↓
DISC 知识库 + Prompt 分析
        ↓
决策层
  - 候选人风格
  - 风险提示
  - 下一步建议
        ↓
详细层
  - D / I / S / C 证据
  - NLP 特征
  - 原始模型输出
```

### 分析模式

当前支持两种模式：

1. **本地规则模式**
   - 永远可用
   - 基于本地解析与规则引擎生成 DISC 分析

2. **LLM 增强模式**
   - 当 `local_settings.py` 中配置了有效 API Key 时启用
   - 采用：
     - `gpt-5-mini`：结构化 transcript
     - `gpt-5.4`：输出更完整的 DISC 风格判断

前端在展示时始终优先突出 **决策层**，而不是原始解释文本。

## 前端设计原则

前端遵循 **三级信息层级**：

### 1. 决策层

首屏直接回答：

- 这个候选人大致是什么风格？
- 有没有风险？
- 接下来该问什么？

### 2. 关键指标层

只保留最重要的判断依据：

- DISC 风格排序
- 真实性 / 风险等级
- 结构能力 / 表达能力

### 3. 详细层

默认折叠，仅给需要深挖的用户：

- D / I / S / C 分维证据
- NLP 指标
- 模型返回细节

这套结构服务的是 **面试官决策路径**，而不是分析展示本身。

## 知识库与 Prompt

DISC 分析依赖于以下两部分：

- `knowledge/DISC.yaml`
- `prompts/disc_system_prompt.txt`

其中：

### `DISC.yaml`

定义了：

- 每个维度的核心动机
- 语言与篇章线索
- 常见误判来源
- 维度区分规则
- 验证性追问建议

### `disc_system_prompt.txt`

定义了：

- 输出 JSON 结构
- 不过度推断的约束
- 面试语境下的分析规则
- 只输出结构化结果的要求

## 样例库

仓库内置了一套经过组织的样例库，用来验证系统在不同场景下的表现，包括：

- 强正向案例
- 中立 baseline 案例
- 普通偏弱案例
- 长文本但空洞的案例
- 高级话术但低信息密度的伪装案例

例如：

- `sales_tob_key_account`
- `backend_order_refactor`
- `operations_growth_content`
- `sales_hollow_long_negative`
- `engineering_shallow_negative`
- `project_polished_low_info`

### 添加新样例

1. 在 `samples/` 下新增一个 `.txt` 文件
2. 在 `samples/index.json` 中补一条索引

建议包含字段：

- `id`
- `title`
- `job_hint`
- `filename`
- `description`

## 本地运行

### 1. 配置本地设置

根据示例创建 `local_settings.py`：

```python
OPENAI_API_KEY = ""
OPENAI_BASE_URL = "https://api.zhizengzeng.com/v1"
OPENAI_PARSER_MODEL = "gpt-5-mini"
OPENAI_ANALYSIS_MODEL = "gpt-5.4"
```

如果未配置 API Key，系统会自动回退到本地规则分析。

### 2. 启动 Demo

```powershell
python run_demo.py
```

然后访问：

```text
http://127.0.0.1:8000
```

## 实际工作方式

### 本地模式

- `app/transcript.py`：解析发言角色与轮次
- `app/features.py`：抽取原子语言特征
- `app/disc_engine.py`：执行本地 DISC 打分

### LLM 模式

- transcript → `gpt-5-mini` 解析
- 结构化问答 + 本地特征 → `gpt-5.4` 高阶分析
- 前端优先展示 LLM 主分析结果，若不可用则回退规则分析

这种分层是有意设计的：

- 先快速结构化
- 再执行更强的高阶推断
- 同时保留稳定的本地 fallback

## 决策哲学

InsightEye 优化的是 **面试官决策**，而不是模型“解释得更多”。

页面真正要帮助用户回答的是：

1. 这个候选人值得继续深入吗？
2. 当前信号可不可信，还是需要验证？
3. 下一步应该追问执行、结构、动机，还是真实性？

因此系统强调：

- 简洁风格总结
- 风险显著提示
- 可直接使用的追问建议

并有意弱化：

- 装饰性图表
- 重复性分析文案
- 首屏出现大量技术细节

---

## 当前限制

当前限制需要被明确理解：

- DISC 推断是概率性的，不是诊断性的
- 长文本仍可能带有自我呈现偏差
- 本地特征提取目前是轻量规则型实现
- LLM 输出质量依赖 transcript 质量与接口稳定性
- 岗位推断是辅助能力，不是最终真值

因此这个仓库应被视为一个 **高潜力原型系统**，而不是成熟的招聘决策引擎。

## 总结

InsightEye 是一个围绕 **AI 原生面试辅助界面** 的早期但严肃的尝试：

- 输入 transcript
- 输出风格 / 风险 / 追问建议
- 优先优化人类判断效率

它真正的价值不在于“展示更多分析”。  
它的价值在于：**用更低的阅读成本，帮助面试官做出更好的判断。**
